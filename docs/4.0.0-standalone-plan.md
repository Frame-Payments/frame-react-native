# Plan: `framepayments-react-native` 4.0.0 — Standalone iOS + Android, 1:1 with Frame iOS 3.0.0 / Android 2.0.9

## Context

`framepayments-react-native` 3.0.1 depends on Frame iOS 3.0.0 and Frame Android 2.0.9. Both native SDKs follow the same pattern: thin REST client over `https://api.framepayments.com` plus a prebuilt UI kit (Checkout + Cart + Onboarding) plus a wallet button (Apple Pay / Google Pay). Every host app inherits Frame-iOS/Android + their transitive deps (Evervault, Sift, FingerprintPro, ProveAuth-via-jfrog, Plaid, PhoneNumberKit / libphonenumber, Frame's Evervault repackages).

**Goal:** ship `4.0.0` as a fully standalone RN package on both platforms with **pixel-perfect 1:1 parity** against today's native SDKs. UI rewritten in TSX. Networking goes through the existing `framepayments` (frame-node) package — but several APIs must be added to frame-node first because the native SDKs use endpoints frame-node doesn't currently expose. Native bridges remain only for things that genuinely require platform primitives: Apple Pay (PassKit), Google Pay (PaymentsClient), DCAppAttestService (iOS) / Play Integrity (Android), Prove, and Sift (only if no usable RN package exists). Breaking release. Both bare RN (≥0.81) and Expo (dev client) supported. Verification is manual side-by-side against current SDKs, sign-off required per screen × state.

## Phase 0 — frame-node API additions (BLOCKING — must land before Phase 1)

frame-node main does not yet expose every endpoint the native SDKs call. These additions are blocking and must ship as a new minor (target 2.1.0) before the RN rewrite can compile.

### Per-call publishable-key auth (cross-cutting)
- Extend the frame-node client constructor to accept `{ secretKey?, publishableKey? }` (one or both).
- Add a per-call option `{ usePublishableKey?: boolean }` on every method that exists in the native SDKs with that flag set. From the iOS source: `ApplePayAPI.*`, `PaymentMethodsAPI.createCardPaymentMethod`, `PaymentMethodsAPI.connectPlaidBankAccount`, `PaymentMethodsAPI.attachPaymentMethod`, `ChargeIntentsAPI.create/confirm`, `TransfersAPI.create`, `CustomerIdentityAPI.create/uploadIdentityDocuments`, `ThreeDSecureVerificationsAPI.create/resend`, `PhoneOTPVerificationAPI.create/confirm`, `GeocomplianceAPI.checkGeocompliance`, `AccountsAPI.getPlaidLinkToken`, `ConfigurationAPI.getEvervault/getSift`, `DeviceAttestationAPI.*`, `WalletAPI.getGooglePayConfiguration`, `SonarSessionsAPI.*`.
- All other methods continue to default to the secret key. RN ships only `pk_*` to the device; host apps continue using `sk_*` server-side.

### New / extended resource modules in frame-node
| Module | Methods to add | Notes |

| `WalletAPI` (new) | `getGooglePayConfiguration()` | Returns `{ environment, processor, processorKey, merchantInfo? }`. pk-key. |
| `ConfigurationAPI` (extend) | `getEvervaultConfiguration()` returning `{ teamId, appId }`, `getSiftConfiguration()` returning `{ accountId, beaconKey, sessionId? }` | Currently only generic `get()`. pk-key both. |

- `GooglePayWalletData` (`{ apiVersion, apiVersionMinor, email?, paymentMethodData }`).

### Tests in frame-node

- One test confirming the constructor accepts publishable-only init and that secret-key methods throw `MissingKeyError` instead of sending the wrong header.
- Multipart test that confirms `uploadIdentityDocuments` serializes `{ uri, type, name }` shapes via axios `FormData`.

- `framepayments@2.1.0` published. Add a `peerDependencies` pin in `framepayments-react-native@4.0.0` to `framepayments@^2.1.0`.

### Phase 6 — Wallet bridges (~1.5 wks, per platform)

#### Apple Pay (iOS) — token-out + two-step settle
- `ios/ApplePayBridge.swift`:
  - `presentApplePay({ amount, currency, applePayMerchantId, requiredBillingFields, supportedNetworks: ['visa','masterCard','amex','discover','JCB'] }) → Promise<{ paymentToken, billingContact, network, transactionIdentifier, displayName?, email? }>`
  - `finishApplePay('success' | 'failure')` settles the still-open sheet after JS finishes the API round-trip — preserves the spinner-stays-up UX.
  - `canMakeApplePay()` — bool, gates the JS `ApplePayButton`.
- JS orchestration in `src/index.ts`:
  ```
  presentApplePay({ owner, amount, currency }):
    guardInitialized(); await attestation.ensureAttested()
    token = await applePayBridge.presentApplePay(...)
    try {
      pm = owner.type === 'customer'
        ? await frame.publishable.applePay.createPaymentMethodWithCustomer({ customerId, paymentToken: token })
        : await frame.publishable.applePay.createPaymentMethodWithAccount({ accountId, paymentToken: token })
      id = owner.type === 'customer'
        ? (await frame.publishable.chargeIntents.create({ amount, currency, customer, paymentMethod: pm.id, confirm: true })).id
        : (await frame.publishable.transfers.create({ amount, accountId, currency, sourcePaymentMethodId: pm.id })).id
      await applePayBridge.finishApplePay('success'); return id
    } catch (e) { await applePayBridge.finishApplePay('failure'); throw e }
  ```

#### Google Pay (Android) — token-out, one-step
- `android/.../GooglePayBridge.kt`:
  - `isGooglePayReady({ environment }): Promise<boolean>` — gates the JS `GooglePayButton`.
  - `presentGooglePay({ amountCents, currencyCode, googlePayMerchantId, environment, walletConfig }) → Promise<{ paymentMethodData, email?, apiVersion, apiVersionMinor }>`
  - Internally builds `PaymentDataRequest` from frame-node's `WalletAPI.getGooglePayConfiguration()` (gateway processor + processor merchant id), allowed networks AMEX/DISCOVER/MASTERCARD/VISA, merchant info, billing address required, emailRequired true. For AddToOwner mode: `totalPriceStatus="NOT_CURRENTLY_KNOWN"`.
  - Rejection codes: `USER_CANCELED`, `GOOGLE_PAY_UNAVAILABLE`, `PAYMENT_FAILED`.
- JS orchestration:
  ```
  presentGooglePay({ owner, amountCents, currencyCode }):
    guardInitialized()
    walletConfig = await frame.publishable.wallet.getGooglePayConfiguration()
    wallet = await googlePayBridge.presentGooglePay({ amountCents, currencyCode, googlePayMerchantId, environment: debugMode ? 'TEST' : 'PRODUCTION', walletConfig })
    return owner.type === 'customer'
      ? (await frame.publishable.chargeIntents.createFromGooglePay({ customerId: owner.id, walletData: wallet })).id
      : (await frame.publishable.transfers.createFromGooglePay({ accountId: owner.id, walletData: wallet })).id
  ```

## Risks / open items

1. **Phase 0 is on the critical path.** The RN rewrite cannot compile against frame-node main as-is. Phase 0 lands in frame-node first, gets cut as `framepayments@2.1.0`, then RN Phase 1 starts. Sequence this with the backend team — several new endpoints (ApplePayAPI, WalletAPI, ConfigurationAPI Evervault+Sift, DeviceAttestation challenge/attest, multipart identity upload, createFromGooglePay variants) need server-side confirmation that they exist and accept publishable-key auth.
2. **Evervault RN SDK availability.** Confirm `@evervault/evervault-react-native` exists in Phase 4 day 1. Fallback to `@evervault/browser` adds ~3 days.
3. **Sift RN package availability.** Phase 9 day 1 audits maintained packages. If none, ship `ios/SiftBridge.swift` + `android/.../SiftBridge.kt`.
4. **Prove on RN.** No public wrapper either platform. Native bridges mandatory. Expo: iOS already has jfrog source in `withFramePodfile.js`; Android adds Maven dep.
5. **Android attestation server-side support.** Frame iOS 3.0.0 ships DCAppAttestService; Android 2.0.9 does not ship Play Integrity. Confirm with backend that `DeviceAttestationAPI.attest` accepts Play Integrity tokens, or scope Android attestation to a follow-up.
6. **`ip_address` request header.** Native SDKs read local interfaces. RN has no clean equivalent. Default: drop the header. Revisit if fraud scoring degrades.
7. **`static_framework = true` removal.** Per the existing podspec history this was for `FramePreloader.h` autolinking export. With Preloader gone, drop it. Restore with new comment only if Phase 11 example builds fail.
8. **Theme tokens missing today.** Current RN bridge silently drops `secondaryButtonText`, `toastBackground`, `toastText`. Phase 2 fixes this and is a behavioral change — flag in CHANGELOG so consumers know toasts now respect theme.
9. **`PaymentCardField` UX.** Frame's native cards do real-time formatting, brand icons, validation. Rebuilding with the same polish is the biggest single-component effort in Phase 2.
10. **Expo Go incompatibility.** Apple Pay, App Attest, Play Integrity, Plaid, Prove, Vision Camera, FingerprintJS Pro, Google Pay PaymentsClient — none work in Expo Go. Document prominently; both examples use dev clients.
11. **Pixel-perfect verification depth.** Manual side-by-side is the chosen approach. Phase 12 budget must absorb a 1.5-week reviewer-driven cycle; if real-world deltas exceed 10/screen, treat as a separate hardening phase before RC.
12. **Doc upload file shape.** RN `FormData` files are `{ uri, type, name }`. Phase 0 must include a frame-node test that proves axios-side multipart serialization works for that shape (the existing `uploadDocuments` requires a pre-uploaded `file_id`, which is fundamentally a different API — net new method).
